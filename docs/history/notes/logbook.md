# Experiment Logbook

## 2026-03-26 — Fix audio corruption during context injection

### Problem

Finetuned model produces corrupted audio: screeching, high-frequency noise, clipping. Two separate root causes were identified.

### Root cause 1: LoRA applied to depformer

`replace_all_linear_with_lora()` in the installed moshi package replaces every `nn.Linear` — including all layers in the **depformer**, the 6-layer transformer that autoregressively generates audio codebook tokens (cb0-cb7 per stream). 43% of all LoRA parameters (107M / 249M) were in the depformer, directly perturbing the pretrained audio generation pathway.

**Diagnosis method:** Per-codebook loss tracking showed agent acoustic codebooks (cb1-7) stuck at ~2.5 throughout training while text loss dropped 42%. Spectral analysis of generated WAVs showed 2295x more high-frequency energy in the finetuned broker channel vs the base model client channel. Corruption worsened progressively from step 32 to step 64.

**Fix:** Added `skip_depformer: bool = True` to `LoraArgs`. When enabled, depformer LoRA parameters are frozen after initialization (`lora_B` is initialized to zeros, so frozen depformer LoRA has zero contribution — depformer behaves identically to pretrained). Only backbone transformer LoRA (142M params) is trained.

**Files:** `moshi-finetune/finetune/args.py`, `moshi-finetune/finetune/wrapped_model.py`

**Result:** Non-injected dialogues (eval_baseline) produced clean audio (high-freq energy dropped from 672 to 0.12). But dialogues with context injection were still corrupted.

### Root cause 2: Train/inference mismatch in context injection audio

The training interleaver (V2 INSERT-SILENCE) and the inference context injection mechanism handle audio channels differently during injection:

| Channel | Training (before fix) | Inference |
|---------|----------------------|-----------|
| Text | `<context>` tokens | `<context>` tokens (forced from queue) |
| Agent audio | SILENCE_TOKENS | Freely generated by depformer |
| User audio | SILENCE_TOKENS | Real audio from other party |

Two mismatches:

1. **Agent audio output** — Training target is silence; inference generates unrestricted audio. Corrupted audio tokens feed back through the autoregressive cache, compounding the error across frames.

2. **User audio input** — Training blanks user audio to silence, but ~70% of injections are proactive (during client speech). At inference, the backbone sees `(context_text + real_user_audio)` which it was never trained on.

**Diagnosis method:** Per-second RMS and spectral analysis of injected vs non-injected dialogues at step 64. Audio during injection frames was quiet (RMS ~0.005) but frames between injections had massively elevated high-freq energy. Mapped `<CTX>` token positions to injection schedule — corruption was concentrated in post-injection windows and worsened with each successive injection.

**Fix (inference):** In `lm.py:process_transformer_output()`, force `sampled_audio_tokens` to `SILENCE_TOKENS` when `_injecting_context` is True. The `provided` mask in the cache update naturally preserves real user audio (channels 9-16 are provided by `input_tokens`), so only agent audio (channels 1-8) is overwritten with silence. This prevents corrupted audio from being decoded and from feeding back.

**Fix (training):** In `interleaver.py`, changed injection insertion to copy the displaced conversation's user audio codes into the insertion block instead of blanking them. Agent audio stays silence (broker should be quiet). This teaches the model: "read context text while hearing the user speak and stay silent" — matching what happens at inference and in production.

**Files:** `personaplex/moshi/moshi/models/lm.py`, `moshi-finetune/finetune/data/interleaver.py`

### Summary of all changes

| File | Change |
|------|--------|
| `finetune/args.py` | Added `skip_depformer: bool = True` to `LoraArgs` |
| `finetune/wrapped_model.py` | Freeze depformer LoRA params when `skip_depformer=True` |
| `finetune/data/interleaver.py` | Preserve user audio at injection frames (don't blank to silence) |
| `personaplex/moshi/moshi/models/lm.py` | Force agent audio to silence during `_injecting_context` |

### Lessons

- **Don't LoRA the depformer.** It's a small (6-layer, 1024-dim) specialized audio codec predictor. LoRA perturbations destroy codebook coherence. Train only backbone LoRA; the backbone learns new text behavior while the frozen depformer preserves audio quality.
- **Training audio must match inference audio during injection.** If training blanks user audio but inference has real audio, the backbone sees an OOD input distribution during injection. Preserve the user audio stream during training injection so the model learns the correct joint distribution.
- **Force silence on agent output during injection at inference.** The model learned to produce silence during context injection, but without explicit enforcement, the autoregressive loop can diverge. One bad frame compounds.
- **Per-codebook loss tracking is essential.** The overall audio loss can look reasonable while individual codebooks (especially acoustic cb1-7) are failing. Agent codebooks (0-7) consistently had 2-3x higher loss than user codebooks (8-15).

## 2026-04-03 — Fix context injection loss due to prompt gap in chunk tiling

### Problem

With `system_prompt.enable: true`, each training chunk prepends a system prompt that steals frames from the conversation window: `conv_frames = num_audio_frames - prompt_length`. But the audio chunking step was still `duration_sec`, meaning each chunk covered less conversation than the step size. This left a "prompt gap" at the tail of each chunk boundary where context injections were silently dropped.

For the pharma-v2 dataset (avg prompt ~18.3s, max ~22.5s at 12.5Hz):

| duration_sec | Injections placed (before) | Injections placed (after) |
|-------------|---------------------------|--------------------------|
| 40 | 3218/5905 (54.5%) | 5905/5905 (100%) |
| 80 | 4517/5905 (76.5%) | 5905/5905 (100%) |

At `duration_sec=80`, nearly **1 in 4 injections was lost** — silently dropped with only a counter increment in `drop_out_of_window`.

### Root cause

`sphn.dataset_jsonl` and the manual chunking loop both stepped by `duration_sec`. But the interleaver only used `conv_frames = num_audio_frames - prompt_length` worth of audio per chunk. An injection with `frame_offset` in the range `[conv_frames, num_audio_frames)` relative to a chunk start fell in the gap and was dropped.

The gap size equals the prompt length (~18s avg, ~22.5s max). More chunk boundaries = more gaps, explaining why smaller `duration_sec` had worse placement rates.

### Fix

Added `prompt_budget_frames` config parameter. When set, the chunk step is reduced to `(num_audio_frames - prompt_budget_frames) / frame_rate`, so conversation windows tile without gaps.

Trade-off: chunks now overlap in audio content by `prompt_budget_frames / frame_rate` seconds. This means ~40% more chunks per epoch (at `duration_sec=80`, step drops from 80s to 57.5s). Some conversation audio is seen twice per epoch with different surrounding context, similar to overlapping-window data augmentation.

The text alignment window was also updated to use `chunk_step_sec` instead of `duration_sec` to prevent text/audio misalignment in the overlap region.

When `prompt_budget_frames=0` (default), behavior is identical to before — fully backward compatible.

### Files changed

| File | Change |
|------|--------|
| `finetune/args.py` | Added `prompt_budget_frames: int = 0` to `SystemPromptArgs` |
| `finetune/data/interleaver.py` | Added `prompt_budget_frames` param to `InterleavedTokenizer`, computes `chunk_step_sec`; text alignment window uses `chunk_step_sec` |
| `finetune/data/dataset.py` | `get_dataset_iterator` and `maybe_load_local_dataset` use `chunk_step_sec` for chunking step |
| `train.py` | Passes `prompt_budget_frames` through; logs effective step at startup |
| `pipeline/compute_prompt_budget.py` | New utility: scans stereo JSONs to compute max prompt frame count |

### Usage

```bash
# 1. Compute budget for your dataset
python pipeline/compute_prompt_budget.py \
  --stereo-dir /path/to/stereo_wav \
  --tokenizer-path /path/to/tokenizer_spm_32k_3.model

# 2. Add to training YAML
system_prompt:
  enable: true
  prompt_budget_frames: 281  # output from step 1
```

### Lessons

- **System prompt cost is not just VRAM** — it creates invisible data loss at chunk boundaries. With a 22.5s max prompt in 80s chunks, 23% of the window is prompt and 23.5% of injections were dropped.
- **Always account for prompt length in chunking stride.** The chunking step should be `duration_sec - prompt_budget`, not `duration_sec`.
- **Injection drop stats (`inj_in_window`, `inj_placed`) increasing with `duration_sec` was the signal** — in a correct implementation these should be constant regardless of chunk size.
