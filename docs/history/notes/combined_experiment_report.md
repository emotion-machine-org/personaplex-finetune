# PersonaPlex Finetuning: Combined Experiment Report

**Date:** April 3--7, 2026
**Model:** PersonaPlex (Moshi 7B variant, `nvidia/personaplex-7b-v1`)
**Domains:** Companion chat, Pharma patient support (Pharma)

---

## Executive Summary

We finetuned PersonaPlex -- a 7B voice-native LLM operating over 17 concurrent channels (text + audio codebooks) at 12.5 Hz -- for two distinct domains: **casual companion conversation** and **structured pharma patient support calls (Pharma)**. Across 25+ experimental runs spanning both domains, we found:

**The core discovery is that learning rate sensitivity dominates everything else.** On our current single-GPU codebase, lr > 1e-6 breaks autoregressive generation quality regardless of regularization, batch size, or chunking strategy. This is true in both domains. An earlier 3-GPU FSDP run succeeded at lr=2e-5, but we cannot reproduce this on 1 GPU -- the mechanism remains an open question.

**Two critical fixes prevent degenerate training:** (1) `text_padding_weight=0.0` eliminates PAD token collapse (discovered in companion, confirmed in pharma), and (2) L2 regularization on `lora_B` weights prevents catastrophic forgetting at low learning rates.

**Current best configs:**

| Domain | LR | Rank | L2 | Pad weight | Batch | Best scores (coh/nat/eff) |
|--------|-----|------|----|------------|-------|---------------------------|
| Companion | 2e-6 | 64 | 1e-3 | 0.0 | 16 | 3.4/3.4/3.4 (v0-22, step 32) |
| Pharma | 1e-6 | 32 | 1e-4 | 0.0 | 16 | 2.2/2.2/3.1 (v2-7, step 192) |

**Key strategic finding:** Companion finetuning is **statistically neutral** vs the base model with a system prompt (n=50 eval). Sampling parameters (`warm_balanced`) yield larger gains (+0.4 avg) than any training change. Pharma finetuning, by contrast, teaches a genuinely new capability (context-injected clinical call handling) that the base model cannot do.

**Currently running:** v2-20 (pharma, lr=1e-6, rank=64, batch=48, 4 epochs) -- stable so far, learning 10% faster than v2-7. ETA ~12 hours from session start.

## 1. Introduction

### PersonaPlex Architecture

PersonaPlex is a 7B-parameter voice model based on the Moshi architecture. It processes three concurrent streams:

| Stream | Channels | Description |
|--------|----------|-------------|
| Text | 1 | Text token predictions |
| Agent audio | 8 | Audio codebook tokens (model's voice) |
| User audio | 8 | Audio codebook tokens (user's voice) |

This yields 17 total channels at a frame rate of 12.5 Hz with a 24 kHz sample rate (Mimi codec). Training uses LoRA on the backbone transformer with the depformer and audio codec layers frozen (`skip_depformer=True`).

### Two Finetuning Targets

**Companion chat:** Casual, open-ended conversation -- empathy, engagement, and active listening. Training data is 1,336 synthetic samples (73.7 hours) generated by Claude and voiced via VibeVoice TTS. The base model already handles this reasonably well with system prompting.

**Pharma/Pharma:** Structured clinical patient support calls with context injection via a "puppeteer" system. The model must learn specific scripts (identity disclosure, DOB verification, AI recording disclosure), handle clinical terminology, and maintain appropriate silence/turn-taking patterns. Training data is 2,003 samples with mean duration 296s. This is a genuinely new capability the base model lacks.

The domains differ in what success looks like: companion needs warmth and naturalness; pharma needs script adherence, clinical accuracy, and structured turn-taking. But they share the same model, architecture, and training infrastructure -- so **the failure modes and fixes discovered in one domain directly transfer to the other**.

## 2. The Core Problem: Catastrophic Forgetting During LoRA Finetuning

Both domains experienced the same fundamental issue: **generation quality degrades as training loss improves**. The model achieves better cross-entropy loss while producing increasingly degenerate output.

### PAD Token Collapse

Discovered in companion experiments, confirmed in pharma. In 180--300 second training chunks at 12.5 Hz, only ~10% of text frames contain actual words. The remaining ~90% are padding tokens. Even with `text_padding_weight=0.1`, padding drove approximately 27% of the effective gradient.

| Step | PAD % | Real text % | Coherence | Text eval loss |
|------|-------|-------------|-----------|----------------|
| 32 | 74.0% | 16.7% | 3.6 | 0.789 |
| 64 | 88.1% | 8.5% | 1.8 | 0.714 |
| 96 | 96.5% | 3.0% | 1.2 | 0.681 |

*(Companion domain data, pre-fix runs)*

The model learned to predict PAD everywhere -- a degenerate optimum that minimizes loss while producing silence.

### The Fix: `text_padding_weight = 0.0`

Setting the padding weight to zero fully masks padding tokens from the text loss gradient. The existing loss computation code already handled this via a `weights > 0.0` guard. **This single change eliminated PAD collapse in all subsequent runs across both domains.**

For pharma, where silence/turn-taking matters more, a small non-zero value (0.03) was later found to be the sweet spot (see Section 5).

### Diagnostics

To monitor this failure mode, the following metrics were added:

- **`real_token_loss`**: Cross-entropy on actual speech tokens only
- **`pad_token_loss`**: Cross-entropy on padding tokens only
- **`real_token_pct`**: Percentage of frames with real tokens
- **`pred_pad_pct`**: Percentage of model predictions that are padding

## 3. L2 Regularization: The Key Innovation

### Implementation

An L2 penalty on LoRA parameters prevents excessive drift from the base model. This was introduced during the pharma v2-7 run and retroactively validated in companion experiments.

### The `lora_A` vs `lora_B` Bug

The first implementation penalized all LoRA parameters. This was incorrect:

- **`lora_A`**: initialized with Kaiming random values (non-zero)
- **`lora_B`**: initialized to zeros
- **Output**: `lora_B @ lora_A * scaling`

Only `lora_B` growing from zero represents actual drift. Penalizing `lora_A` wasted gradient budget fighting random initialization, producing a constant ~0.289 L2 loss component that never changed.

**Confirmation:** Companion run v0-15 (broken L2 on all params) performed identically to v0-14 (no L2). After fixing to target `lora_B` only, regularization worked as intended.

### Results Across Both Domains

**Companion:** v0-17 (lr=1e-6, rank=32, L2=1e-3 fixed) sustained scores of ~2.5--2.9 post-peak instead of collapsing to 1.0. v0-22 (lr=2e-6, rank=64, L2=1e-3) achieved the best overall companion score of 3.4.

**Pharma:** v2-7 (lr=1e-6, rank=32, L2=1e-4) was the first run where metrics stayed above baseline through full training. Without L2, every prior pharma run degraded.

### Interaction with Learning Rate

L2 cannot compensate for excessive learning rate. Companion v0-28 (lr=1e-5, L2=5e-3) collapsed to 1.0 despite 5x stronger regularization. Pharma v2-8 (lr=5e-6, L2=4e-4) collapsed by step 64. **L2 stabilizes learning; it does not make aggressive learning rates safe.**

## 4. Learning Rate Sensitivity

**This is the single most important finding across both domains: lr > 1e-6 breaks generation with the current single-GPU codebase.**

### Evidence From Both Domains

**Companion domain:**

| LR | Best outcome |
|----|-------------|
| 5e-7 | Too slow, declined from start (v0-18) |
| 1e-6 | Stable, best single-seed score 3.5 (v0-16) |
| **2e-6** | **Best config overall, score 3.4 (v0-22)** |
| 4e-6 | Too aggressive, declined (v0-23) |
| 5e-6 | Killed early, collapsed (v0-25) |
| 1e-5 | Collapsed despite strong L2 (v0-28) |

**Pharma domain:**

| LR | Best outcome |
|----|-------------|
| **1e-6** | **Only stable option; 2.2/2.2/3.1 at step 192 (v2-7)** |
| 5e-6 | Transient good window at step 64, collapsed by step 128 (v2-13, v2-14) |
| 1e-5 | Collapsed immediately, degenerate repetition (v2-11) |
| 2e-5 | Collapsed even with old chunking and batch=48 (v2-16) |

The pharma domain is **more sensitive** than companion, with the stability boundary at 1e-6 rather than 2e-6. This may reflect the longer sample durations (300s vs 180s) or the more structured nature of the target behavior.

### The Old 3-GPU Mystery

The original pharma run (March 31) succeeded at lr=2e-5 on 3 GPUs with FSDP. This cannot be reproduced on 1 GPU. Tested hypotheses:

| Hypothesis | Test | Result |
|-----------|------|--------|
| Old chunking strategy | v2-16: old chunking + lr=2e-5 + batch=48 | **Rejected** -- collapsed at step 32 |
| Larger batch size | v2-15: batch=48 + lr=5e-6 | **Rejected** -- delayed but did not prevent collapse |
| FSDP gradient averaging | Not directly testable | **Leading hypothesis** -- FSDP may reduce effective gradient magnitude |

### The Transient Good Generation Window

At lr=5e-6, pharma models pass through a narrow window of good generation around step 64 before collapsing. This suggests the optimization landscape has a useful region that higher learning rates overshoot. Companion experiments show a similar pattern at lr=4e-6.

## 5. Hyperparameter Effects

### LoRA Rank

**Higher rank = faster memorization, not better generalization.** Confirmed in both domains.

| Domain | Comparison | Finding |
|--------|-----------|---------|
| Companion | Rank 128 vs 64 vs 32 | Rank 128 peaks sooner, overfits faster (v0-26 vs v0-22) |
| Pharma | Rank 64 vs 32 at lr=1e-6 | Rank 64 learns 10% faster (eval loss 0.960 vs 1.078 at step 64) without stability issues |

At stable learning rates, rank 64 is preferred over 32 for the faster convergence without instability.

### Batch Size

Larger batches produce smoother gradients but **do not change the LR stability boundary**. Pharma v2-15 (batch=48, lr=5e-6) delayed collapse by roughly one eval step compared to v2-14 (batch=16), but still collapsed.

### text_padding_weight

Controls silence and turn-taking behavior. The optimal value depends on the domain:

| Value | Effect |
|-------|--------|
| 0.0 | No silence signal -- model chooses its own pattern. Best for companion. |
| 0.01 | Too weak at high LR (pharma v2-13) |
| **0.03** | **Good balance for pharma -- natural turn-taking without overcorrection** |
| 0.05 | Overcorrects, producing 82% silence (pharma v2-12) |
| 0.1 | Causes PAD collapse in long training runs |
| 0.5 | Severe silence overfit (old pharma run) |

### audio_loss_weight

**Neutral for generation quality.** Companion v0-29 (text-only, audio_loss=0.0) produced the same quality as joint training at 2x the speed. Audio loss does not appear to contribute to or detract from text quality in either domain.

### Duration and Chunking

The old pharma run used 80s chunks with the old chunking strategy (`chunk_step = duration_sec`, no `prompt_budget_frames`). The new code uses longer chunks with prompt budgets. **Direct testing (v2-16) confirmed chunking strategy is NOT the factor** explaining the old run's success.

## 6. Domain-Specific Findings

### 6.1 Companion Domain

**Finetuning is statistically neutral vs the base model.** A 50-dialogue multi-seed evaluation (10 seeds x 5 scenarios) showed no meaningful difference:

| Model | Coherence | Naturalness | Effectiveness |
|-------|-----------|-------------|---------------|
| Base | 3.34 +/- 1.09 | 3.30 +/- 0.98 | 2.86 +/- 1.11 |
| Finetuned (v0-16, step 96) | 3.22 +/- 1.08 | 3.28 +/- 0.87 | 2.92 +/- 1.06 |

The base model with a companion system prompt already captures the target behavior.

**Sampling parameters matter more than training.** The `warm_balanced` config (temp_text=0.7, topk_text=30, temp_audio=0.55, topk_audio=100) produced a **+0.4 average improvement** over baseline sampling -- larger than any training change.

**Text ceiling test confirms the eval pipeline works:** ideal Claude-generated conversations score 5.0/4.0/5.0 (coherence/naturalness/effectiveness). The 4.0 naturalness ceiling reflects inherent limitations of text-only conversations.

**Repetition penalty:** Marginal gain at 1.2, actively harmful above. The model's repetition problem is phrase-level ("I think it's, I think it's"), not token-level, so token-level penalties cannot address it.

**Qualitative failure modes:**
1. Repetitive phrase loops, especially on emotional/open-ended topics
2. Fabrication of personal experiences and facts
3. Excessive hedging with filler words on emotional topics
4. Casual/open-ended topics are hardest; structured emotional topics work best

### 6.2 Pharma/Pharma Domain

**Context injection (puppeteer) makes this a genuinely new capability** that the base model cannot perform. Unlike companion chat, pharma finetuning has real value.

**The old successful run** (March 31, 3 GPUs) demonstrated the model can learn specific scripts: "Hi, this is Mia, I'm an automated care assistant", DOB verification, and recording disclosure. It reached 3.0/3.0/4.0 at step 1024. However, it suffered from silence overfit (text_padding_weight=0.5) and **cannot be reproduced on the current single-GPU codebase**.

**v2-7 (lr=1e-6, rank=32):** The best current run. Stable throughout training, with best scores of 2.2/2.2/3.1 at step 192. Did NOT learn specific scripts -- the learning rate may be too low for memorizing exact phrases.

**v2-20 (lr=1e-6, rank=64, batch=48, 4 epochs):** Currently running. At step 64, text_eval_loss is 0.960 vs v2-7's 1.078 -- **10% faster learning** from rank=64. Stable so far. This is the most promising active run.

**Silence percentage is a critical domain-specific metric.** Pharma calls require natural turn-taking -- too little silence (7%, v2-11) means the model talks non-stop; too much (82%, v2-12) means it goes mute. The 60--70% range at text_padding_weight=0.03 appears appropriate.

## 9. Conclusions

Findings confirmed across **both domains** are marked with (B). Domain-specific findings are marked (C) for companion or (A) for pharma.

1. **(B) `text_padding_weight` must be carefully tuned to prevent PAD collapse.** Setting it to 0.0 is essential as a starting point. For domains requiring silence/turn-taking control, 0.03 is the sweet spot.

2. **(B) L2 regularization on `lora_B` only is essential for stable training.** Without it, all runs degrade. Penalizing `lora_A` is a no-op due to Kaiming initialization.

3. **(B) Learning rate > 1e-6 breaks generation on single-GPU training.** Higher LR produces better cross-entropy loss but degenerate output (repetition loops, silence collapse). The companion domain tolerates up to 2e-6; pharma's stability boundary is 1e-6.

4. **(B) Higher LoRA rank accelerates memorization, not generalization.** Rank 64 learns ~10% faster than rank 32 at stable LRs without instability. Rank 128 overfits faster.

5. **(B) Batch size smooths gradients but does not change the LR stability boundary.** Batch=48 delays collapse by ~1 eval step at unstable LRs.

6. **(B) L2 cannot compensate for excessive learning rate.** lr=1e-5 with L2=5e-3 still collapses (companion v0-28). lr=5e-6 with L2=4e-4 still collapses (pharma v2-8).

7. **(B) Audio loss is neutral for text generation quality.** Text-only training produces equivalent quality at 2x speed (companion v0-29).

8. **(C) Companion finetuning is statistically neutral vs base model + system prompt** (n=50 multi-seed eval). The base model already captures companion behavior.

9. **(C) Sampling parameters matter more than training for companion.** `warm_balanced` (temp_text=0.7, topk_text=30) yields +0.4 avg -- larger than any training change.

10. **(C) The quality gap (3.3 vs 5.0 ceiling) is driven by phrase-level repetition, fabrication, and hedging** -- none addressable by token-level repetition penalty.

11. **(A) The old 3-GPU pharma run's success at lr=2e-5 is NOT explained by chunking strategy** (directly tested and rejected). The leading hypothesis is FSDP gradient averaging.

12. **(A) At higher LRs, the model passes through a transient "good generation window"** (~step 64 at lr=5e-6) before collapsing, suggesting a narrow useful region in parameter space.

13. **(A) Context injection (puppeteer) represents a genuinely new capability** that justifies finetuning -- unlike companion, where the base model is already sufficient.

14. **(A) Silence percentage is a critical pharma-specific metric.** 0.03 padding weight maintains natural 60--70% silence for clinical turn-taking.

## 10. Recommended Next Steps

### High Priority

1. **Wait for v2-20 results.** This is the most promising pharma run (lr=1e-6, rank=64, batch=48, 4 epochs). If it learns specific scripts at step 192+, the lr=1e-6 approach is validated.

2. **Investigate FSDP gradient behavior.** Understanding why 3-GPU FSDP allowed lr=2e-5 could unlock 20x faster effective learning rates. Profile gradient magnitudes under FSDP vs single-GPU.

3. **Staged warmup strategy.** Train at lr=1e-6 to a stable checkpoint, then carefully increase to lr=5e-6 to exploit the "good generation window" without overshooting.

4. **Adopt `warm_balanced` sampling as default** for all evaluation and production: temp_text=0.7, topk_text=30, temp_audio=0.55, topk_audio=100.

### Medium Priority

5. **Consider DPO instead of SFT.** Preference learning may generalize better, particularly for subtle conversational qualities (companion) and script adherence (pharma).

6. **Improve companion training data quality.** Remove fabrication patterns; add more nuanced emotional responses with active listening rather than generic advice.

7. **Separate LoRA adapters for text vs audio channels.** May resolve the tension between text and audio quality during training.

### Exploratory

8. **Gradient accumulation to simulate multi-GPU.** If FSDP gradient averaging is the mechanism, gradient accumulation with appropriate scaling might recover the higher-LR regime.

9. **Curriculum learning for pharma.** Start with short, simple script segments before full 5-minute calls.

10. **Repetition penalty at the phrase level.** Token-level penalties are ineffective; a bigram or trigram penalty may address the actual failure mode.
