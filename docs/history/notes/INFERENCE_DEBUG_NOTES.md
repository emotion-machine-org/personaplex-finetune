# Inference Debugging Notes — Finetuned PersonaPlex on RTX 4090

**Date:** 2025-04-03
**Machine:** 5x RTX 4090 (24GB each), Tailscale network
**Model:** pharma-v2 step 1024 (LoRA r64, merged weights)
**Server:** `personaplex/moshi/moshi/server.py` + `personaplex/moshi/moshi/models/lm.py`

---

## What Works

- **Base PersonaPlex model** (no LoRA) works perfectly in the inference UI — voice flows both ways, model responds naturally.
- **Finetuned model in gen_eval** (bot-to-bot offline evaluation) produces high-quality, coherent dialogue.
- **Connection/WebSocket infrastructure** is solid — SSL via self-signed cert on Tailscale IP, sphn 0.1.12 for Opus encoding.

## What Doesn't Work

Finetuned model in the real-time WebSocket UI produces clunky, unreliable output — late responses, frequent silence, occasional screech.

---

## Root Causes Identified

### 1. Voice Prompt `.pt` Files Are Incompatible with Finetuned Models

**The biggest issue.** The `.pt` voice prompts (e.g., `NATF0.pt`) from `nvidia/personaplex-7b-v1` contain pre-saved embeddings AND a token cache generated by the **base model**. When loaded into a finetuned model:

- `_step_voice_prompt_core()` replays the embeddings through the finetuned model's transformer (building a finetuned KV cache)
- Then `state.cache.copy_(self.voice_prompt_cache)` overwrites the token cache with **base model tokens**
- This mismatch between the KV cache (finetuned) and token cache (base) causes the text channel to collapse to PAD with 97-99% probability

**Evidence:** With voice prompt skipped, PAD probability drops to 48% by step 12 and the model generates text. With voice prompt loaded, PAD stays at 97%+ for 100+ steps.

**Fix options:**
- Skip `.pt` voice prompts for finetuned models (current workaround)
- Re-generate `.pt` files using the finetuned model (`save_voice_prompt_embeddings=True`)
- Use raw `.wav` voice prompts instead of `.pt` (let the finetuned model build its own cache)

### 2. Sampling Parameters Were Not Applied

Server lines 146-149 were **commented out**:
```python
# self.lm_gen.temp = float(request.query["audio_temperature"])
# self.lm_gen.temp_text = float(request.query["text_temperature"])
# ...
```

So the model always used LMGen defaults (temp=0.8, topk=250) regardless of UI settings. Gen_eval uses much more conservative values (temp_audio=0.45, temp_text=0.4, topk_audio=80, topk_text=15).

**Fix:** Uncomment these lines. Done in current working tree.

### 3. Greeting Mechanism Deadlocks with Finetuned Models

The original greeting logic only fires when the model naturally produces a non-PAD text token:
```python
if self._greeting_queue:
    if sampled_id not in (0, 3):  # Only when model wants to speak
        forced_id = self._greeting_queue.popleft()
```

Since the finetuned model is biased toward PAD (especially without a compatible voice prompt), the greeting **never fires** — the model stays silent forever.

**Fix:** Force greeting tokens unconditionally. Done in current working tree.

### 4. System Prompt Processing Takes Too Long

The text prompt (193 tokens for the Lipitor adherence prompt) takes ~16 seconds to process (one model forward pass per token at 12.5 Hz). During this time:
- The original `is_alive()` function consumes WebSocket messages (audio from client is lost)
- If handshake is sent after system prompts, the client's 10-second inactivity timeout kills the connection
- If handshake is sent before system prompts, the client gets no audio for 16 seconds and the inactivity timeout still kills it

**Note:** This was NOT an issue with the base model because voice prompt `.pt` replay is fast (just embedding stepping) and the overall system prompt time was under 10 seconds.

**Fix options:**
- Truncate text prompt to ~80 tokens (keeps processing under 10 seconds)
- Increase client-side inactivity timeout
- Send keep-alive silence during system prompt processing (attempted — hard because system prompts block the event loop)
- Process system prompts in a background thread (attempted — CUDA + threading issues)

### 5. Heavy PAD Bias in Finetuned Model

Even with all infrastructure fixes, the finetuned model has a strong PAD bias:
- Steps 0-100: PAD probability 96-99%
- Steps 100-150: PAD gradually drops to 30-50%
- Steps 150+: Model occasionally generates text, then goes back to PAD

This is likely caused by `text_padding_weight: 0.5` in training — the model learned to be silent 50% of the time. In gen_eval this works because bot-to-bot provides clean, deterministic audio input. With real mic audio (noise, Opus compression, network jitter), the model defaults to silence.

**Fix options:**
- Apply a PAD logit penalty at inference time (subtract a bias from PAD logit before sampling)
- Retrain with lower `text_padding_weight` (e.g., 0.05-0.1)
- Use the `inject_continuation` mechanism (nudge) to break silence deadlocks

### 6. `sphn` Version Incompatibility

The main venv had `sphn 0.2.1` which changed the API:
- `OpusStreamReader.read_pcm()` → removed in 0.2
- `OpusStreamWriter.read_bytes()` → removed in 0.2

**Fix:** Downgrade to `sphn>=0.1.4,<0.2`. Done: installed `sphn 0.1.12`.

## Key Architecture Reference

- Moshi: 3-stream model — text (1ch) + agent audio (8 codebooks) + user audio (8 codebooks) = 17 channels
- Frame rate: 12.5 Hz (Mimi codec), sample rate 24kHz
- `dep_q=16` for PersonaPlex (vs standard Moshi dep_q=8)
- Server decodes tokens[:, 1:9] for agent audio output
- System prompt order: voice prompt → silence (6 frames) → text prompt → silence (6 frames)
- Greeting: queued text tokens, forced during early inference steps
- Gen_eval sampling: temp_audio=0.45, temp_text=0.4, topk_audio=80, topk_text=15
